    控制反转与依赖注入

    本文参考自 Martinfowler 的文章，链接如下。

    Inversion of Control Containers and the Dependency Injection pattern

    从最我们常见的使用方式来讲，什么样的控制被反转了？

    考虑一个我们常见的场景：

    在这个场景下，AccountController 依赖 IAccountService 获取账户信息。IAccountService 在构造器中被实例化出来。

    而过程中 Controller 即控制了 IService 的实例化，又使用它获取数据。这导致 Controller 既要依赖 IService 接口，同时也要依赖 Service 的实现。

    关于这种场景的弊端，老马的文章中有很详细的例子，此处不再多做赘述，主要是强耦合带来的问题。我们主要来探讨 IoC 是怎么样解开这种耦合的。

    以上是一种常见的构造器注入方式，IAccountService 的实例化不再由 Controller 自己管理，而是由另一个独立的对象来负责，我们可以称其为装配器 Assembler，现代框架一般称其为容器 Container。

    可以简单理解为，容器构造了 IAccountService 的实例，并将其传入到 Controller，因此 Controller 只需要使用即可，而不需要知道它是如何被构造出来的。

    回到一开始的问题，什么样的控制被反转了？

    答案是 AccountController 对它所依赖的 IAccountService 的获取被反转了。从一开始的主动实例化，变成了从容器中获取，而容器负责对其依赖的构造以及生命周期管理。

    那依赖注入又是什么？

    老马认为控制反转这个词并不能精确的描述出上文中这种依赖的处理方式，他举了一个用户界面的例子，我这里给出另一个相似的例子，早期的前端开发大都是手动控制 DOM 渲染，比如 PHP、JSP，JQuery 等，而现代的 UI 框架则是从数据出发，开发者操纵数据，框架监听数据变化并生成 DOM 以及渲染。可以理解为对渲染的控制反转。

    由此，他们提出了一个新的词语用来描述这种依赖间的关系，这就是依赖注入 Dependency Injection。

    服务依赖的具体实现由容器注入，服务与依赖的关系由容器管理。
